# 纯文字转网页文章 Markdown 操作规范

本规范用于将纯文字草稿整理为本项目可直接发布的 Markdown 文章。

适用范围：

- 文章目录：`src/content/posts/`
- 文件格式：`.md`
- 站点类型：Astro content collection
- 发布方式：写入文章文件后提交到 GitHub 并触发自动部署

---

## 1. 目标

当你手里只有一段或多段纯文字时，本规范帮助你把它整理成：

- 符合本项目 schema 的文章文件
- 能正常出现在首页时间线与文章页中的内容
- 具备统一的标题、摘要、标签、阅读时间和正文结构

---

## 2. 输出文件位置与命名

每篇文章都保存到：

```text
src/content/posts/
```

文件命名规则：

```text
case-001.md
case-008.md
case-120.md
```

要求：

- 文件名统一使用小写
- 前缀固定为 `case-`
- 序号建议使用三位数字
- 文件名中的序号应与 `caseId` 保持一致

示例：

- 文件名：`case-008.md`
- 对应 `caseId`：`CASE-008`

---

## 3. 每篇文章的标准结构

每篇文章由两部分组成：

1. Frontmatter
2. 正文内容

标准模板如下：

```md
---
caseId: "CASE-008"
date: "2026.06.01"
tags: ["读书", "随笔"]
readTime: "5 MIN"
title: "我看完了《人类简史》"
subtitle: "第三遍读，这次看到了别的东西。"
draft: false
---

这里开始写正文。
```

---

## 4. Frontmatter 字段规范

以下字段必须按本项目现有规则填写。

### 4.1 `caseId`

- 类型：字符串
- 必填
- 格式：`CASE-001`

规则：

- 必须大写
- 固定前缀为 `CASE-`
- 序号建议与文件名一致

### 4.2 `date`

- 类型：字符串
- 必填
- 格式：`YYYY.MM.DD`

正确示例：

```text
2026.06.01
2025.12.09
```

错误示例：

```text
2026-06-01
2026.6.1
2026/06/01
```

注意：

- 月份和日期必须补零
- 本项目首页时间线按字符串倒序排序，所以日期格式必须严格统一

### 4.3 `tags`

- 类型：字符串数组
- 必填

正确示例：

```yaml
tags: ["市场", "读书"]
```

错误示例：

```yaml
tags: "市场,读书"
tags: 市场
```

规则：

- 使用 1 到 4 个标签即可
- 标签尽量短，不要写成长句
- 保持稳定命名，避免同义词混用

推荐标签示例：

- `市场`
- `读书`
- `历史`
- `随笔`
- `复盘`
- `meta`

### 4.4 `readTime`

- 类型：字符串
- 必填
- 推荐格式：`5 MIN`

规则：

- 使用整数分钟
- 统一大写 `MIN`

参考：

- 短文：`2 MIN` 到 `4 MIN`
- 正常文章：`5 MIN` 到 `10 MIN`
- 长文：`12 MIN` 以上

### 4.5 `title`

- 类型：字符串
- 必填

规则：

- 用于文章页主标题和首页卡片标题
- 应直接、清晰、可读
- 不要过长，尽量控制在一行或两行内能读完

### 4.6 `subtitle`

- 类型：字符串
- 必填

规则：

- 用作文章摘要和导语
- 长度建议为 1 到 2 句
- 用于首页预览时，应能概括文章核心

### 4.7 `draft`

- 类型：布尔值
- 可选，默认 `false`

规则：

- `draft: true` 表示草稿，不会被发布
- `draft: false` 表示正常发布
- 建议每篇文章显式写出该字段，避免误判

## 5. 正文转换规则

当原始材料是纯文字时，正文整理遵循以下规则。

### 5.1 段落处理

规则：

- 一个完整意思写成一个自然段
- 段落之间空一行
- 不要为了视觉效果手动把一句话拆成很多行

正确示例：

```md
今天重新看了一遍自己的交易记录，发现真正让我亏损的不是判断错误，而是没有执行自己的纪律。

这件事其实并不新鲜，只是每次复盘时都需要重新承认一次。
```

不推荐示例：

```md
今天重新看了一遍自己的交易记录，
发现真正让我亏损的不是判断错误，
而是没有执行自己的纪律。
```

### 5.2 小标题处理

当纯文字内容存在明显分段时，可补充标题。

推荐层级：

- `##` 作为主分节标题
- `###` 作为子分节标题

说明：

- 不使用 `#` 作为正文标题
- 因为页面主标题已经由 frontmatter 中的 `title` 提供

示例：

```md
## 我为什么又复盘了一次

正文段落。

## 这次看到的真正问题

正文段落。
```

### 5.3 列表处理

当原始文字包含并列观点、步骤、条目时，转换为列表。

无序列表示例：

```md
- 我判断对了方向
- 我没有控制仓位
- 我提前承认了错误
```

有序列表示例：

```md
1. 先记录事实
2. 再解释原因
3. 最后总结教训
```

### 5.4 强调处理

用于突出重点词句。

规则：

- 关键词用 `**加粗**`
- 语气、书名或轻强调内容用 `*斜体*`
- 不要连续多句都加粗

示例：

```md
真正重要的不是预测，而是**执行纪律**。

我最近重读了*资治通鉴*。
```

### 5.5 引用处理

当正文出现引用、摘录、原话时，使用 Markdown 引用。

示例：

```md
> 市场最擅长教育那些以为自己已经学会的人。
```

## 6. 本站可用的增强样式

本项目支持在 Markdown 正文中直接插入少量 HTML，以获得更贴合页面风格的展示效果。

### 6.1 `pullquote`

用于强调一句关键的话。

写法：

```html
<div class="pullquote">这里是一句值得被单独展示的话。</div>
```

适用场景：

- 开篇引句
- 段落中最重要的结论
- 有节奏感的单句表达

使用原则：

- 一篇文章建议 1 到 3 处
- 不要把长段落塞进去

### 6.2 `code-block`

用于展示终端感、记录感、命令感的内容。

写法：

```html
<div class="code-block">BUY 0.5 BTC @ 62000
STOP LOSS @ 59000</div>
```

适用场景：

- 交易记录
- 观察日志
- 简短命令或文本片段

### 6.3 `figure-block`

用于图片和配文说明。

写法：

```html
<div class="figure-block">
  <img src="/images/posts/case-008/chart-1.png" alt="图表说明" />
  <div class="caption">图 1：某一段时间的价格变化。</div>
</div>
```

适用场景：

- 截图
- 图表
- 书页照片
- 注释图片

## 7. 图片与静态资源放置规则

如果文章需要配图，推荐放在：

```text
public/images/posts/case-008/
```

引用方式：

```text
/images/posts/case-008/chart-1.png
```

规则：

- 图片文件放在 `public/` 下
- Markdown 或 HTML 中引用时使用以 `/` 开头的站点路径
- 文件名尽量语义化

推荐命名：

- `cover.jpg`
- `chart-1.png`
- `note-quote.png`

## 8. 纯文字转 Markdown 的具体步骤

### 步骤 1：先补齐文章元信息

在开始整理正文前，先确定：

- `caseId`
- `date`
- `tags`
- `readTime`
- `title`
- `subtitle`
- `draft`

如果原稿没有这些信息，需手动补充。

### 步骤 2：将原始文字整理为自然段

处理原则：

- 去掉无意义断行
- 把语义连贯的句子合并成段
- 每段只表达一个主要意思

### 步骤 3：识别结构并补小标题

如果原稿内容包含：

- 开头引入
- 中间论述
- 结尾总结

则可用 `##` 把结构明确出来。

### 步骤 4：提取重点句

当正文中出现特别有力的一句时：

- 可以保留在原段中
- 也可以单独转成 `pullquote`

### 步骤 5：把并列信息改成列表

适用于：

- 原因
- 结论
- 观察项
- 操作步骤

### 步骤 6：完成后统一检查 frontmatter 与格式

重点检查：

- 字段是否齐全
- 日期格式是否正确
- 标签是否为数组
- 段落之间是否空行
- 是否误写了 `draft: true`

## 9. 建议的发稿输入格式

当你把纯文字交给我整理时，建议按下面格式提供。

```text
caseId: CASE-008
date: 2026.06.01
tags: 读书, 随笔
readTime: 5 MIN
title: 我看完了《人类简史》
subtitle: 第三遍读，这次看到了别的东西。
draft: false

正文：
这里粘贴你的纯文字草稿。
可以是一整段，也可以是很多段。
```

如果有额外要求，也可以一并说明：

- 哪一句想单独做成引言
- 是否要加小标题
- 是否保留口语感
- 是否要更克制或更文学

## 10. 一份可直接参考的成品示例

```md
---
caseId: "CASE-008"
date: "2026.06.01"
tags: ["读书", "随笔"]
readTime: "5 MIN"
title: "我看完了《人类简史》"
subtitle: "第三遍读，这次看到了别的东西。"
draft: false
---

这本书我已经不是第一次读了，但这次重读时，注意力不再放在那些宏大的结论上，而是放在它讲故事的方式上。

<div class="pullquote">很多时候，我们以为自己是在理解历史，其实只是在寻找能安放自己立场的句子。</div>

## 我这次真正看到的东西

以前读它，我在意的是观点。现在再读，我更在意它怎样把复杂问题讲成一个让人相信的故事。

这让我意识到，写作者真正厉害的地方，往往不是答案，而是组织答案的方式。

## 这和我有什么关系

我后来想，市场也是一样。

- 信息很多
- 解释很多
- 观点很多
- 但真正有价值的是能不能持续面对事实

这件事并不新鲜，只是每次都要重新学一次。
```

## 11. 发布前检查清单

发布前逐项确认：

- 文件位于 `src/content/posts/`
- 文件名符合 `case-XXX.md`
- `caseId` 与文件名对应
- `date` 格式为 `YYYY.MM.DD`
- `tags` 为字符串数组
- `title` 与 `subtitle` 已填写
- `draft` 为 `false`
- 正文段落之间有空行
- 如果使用图片，资源已放入 `public/`

## 12. 常见错误

### 错误 1：日期格式不统一

错误：

```yaml
date: "2026-6-1"
```

正确：

```yaml
date: "2026.06.01"
```

### 错误 2：标签写成字符串

错误：

```yaml
tags: "读书,随笔"
```

正确：

```yaml
tags: ["读书", "随笔"]
```

### 错误 3：缺少必填字段

错误示例：

- 没有 `subtitle`
- 没有 `readTime`
- 没有 `title`

### 错误 4：文章放错目录

错误目录：

```text
src/posts/
content/posts/
```

正确目录：

```text
src/content/posts/
```

### 错误 5：草稿未取消

错误：

```yaml
draft: true
```

结果：

- 本地可能能看到文件
- 但不会正常出现在发布后的文章列表中

## 13. 执行口径

今后当“纯文字转 Markdown”任务由我代为处理时，默认执行口径如下：

- 优先保持原文意思不变
- 优先整理结构，不擅自扩写观点
- 自动补齐符合本项目要求的 frontmatter
- 自动把散乱断句整理为自然段
- 需要时补充小标题、列表、强调和 `pullquote`
- 若原稿信息不足，会先向你追问缺失字段

如无特别说明，本规范长期作为本项目文章整理标准使用。
